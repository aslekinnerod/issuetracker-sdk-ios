# Issuetracker SDK — iOS sample app

A real installable iOS app that exercises every public surface of
the SDK so you can shake-test changes in 30 seconds. Same feature
checklist as the Android, Flutter, and React Native sample apps —
keep them in lockstep when adding capabilities.

## How to run

1. Install [XcodeGen](https://github.com/yonaskolb/XcodeGen) if you
   don't have it (`brew install xcodegen`). The `.xcodeproj` is
   regenerated from `project.yml`; checking the project file in
   would produce daily pbxproj-merge conflicts and we don't want
   that.

2. Generate the Xcode project:

   ```
   cd sdk-ios/example-app
   xcodegen generate
   ```

3. Open `IssuetrackerExample/IssuetrackerExampleApp.swift` and
   replace the `API_KEY` constant with a real key from the
   Issuetracker admin UI. (Or copy that file to
   `IssuetrackerExampleApp.local.swift`, gitignore it, and edit
   there for day-to-day work.)

   If you skip this step the build still works but the SDK will
   fall into `invalid_api_key` → TERMINATED on the first report —
   a useful demo path, just not the happy one.

4. Open the generated project and run:

   ```
   open IssuetrackerExample.xcodeproj
   ```

   Pick a simulator (iPhone 16+ on iOS 17+ recommended) and Cmd-R.
   Or from the command line:

   ```
   xcodebuild -project IssuetrackerExample.xcodeproj \
     -scheme IssuetrackerExample \
     -destination 'platform=iOS Simulator,name=iPhone 16,OS=26.0' \
     build
   ```

## Feature checklist

Each section in the app maps 1:1 to a surface in the SDK. When you
add a feature, add a section here AND in the sister sample apps so
the four platforms stay comparable.

| Section | What it exercises | SDK API |
|---|---|---|
| **Lifecycle** | Shows the last `onConfigurationError` reason the SDK reported. Has a "Reset" button so you can retry a path that has already fired. | `onConfigurationError` callback on `configure(...)` |
| **Reporting** | One button programmatically opens the reporter. Shake-to-report and two-finger long-press also trigger it. | `Issuetracker.report()` + `shakeToReport` + `longPressToReport` |
| **Identity** | Sets / clears the display name that stamps every report. | `identify(name:)` / `clearIdentity()` |
| **Breadcrumbs** | Records up to two action breadcrumbs with optional metadata. The most-recent 5 ride along on every report. | `recordAction(_:metadata:)` |
| **Onboarding** | Re-presents the first-launch trigger introduction popover regardless of whether it has been shown before. | `Issuetracker.showOnboarding()` |
| **TERMINATED-UI i18n** | Toggle a Norwegian translation of the terminal-state strings. Restart the app to apply (`configure(...)` runs once in `App.init`). | `terminatedUI: TerminatedUiStrings` on `configure(...)` |
| **Destructive** | Crash-test button (with confirmation). The next launch picks up the crash and queues a report. | `Issuetracker._testCrash()` |

## Folder layout

```
example-app/
  README.md                              ← you are here
  project.yml                            ← XcodeGen spec — source of truth
  .gitignore                             ← ignores generated .xcodeproj + DerivedData
  IssuetrackerExample/
    IssuetrackerExampleApp.swift         ← @main App + Issuetracker.configure()
    ContentView.swift                    ← single screen + one View per section
    Info.plist                           ← generated by XcodeGen from project.yml
```

The companion sample apps mirror this layout per platform convention:

- `sdk-android/sample-app/` (Compose single screen)
- `sdk-flutter/example/` (Material 3 single screen)
- `sdk-react-native/example/` (Pressable cards single screen)
- `sdk-web/example/` (existing — extend to match this feature set)

Each sample-app README is the same shape (How to run + Feature
checklist table) so a person can hop between platforms and find the
same affordances.

## Adding a new feature

When the SDK gains a new public surface:

1. Add a section to **this** app (a private `XxxSection: View` in
   `ContentView.swift`).
2. Add the row to the **Feature checklist** table above.
3. Replicate to the other three sample apps using the same section
   title + same affordance (button label / toggle label).

## What this app intentionally is NOT

- Not a production reference app — uses `UserDefaults` for tiny
  state (i18n toggle + last error display), not Core Data / SwiftData.
- Not a layout exemplar — single scroll, system-grouped-background
  cards. The point is to surface every SDK affordance, not to look
  pretty.
- Not multi-screen — keeps the surface flat so anyone testing the
  SDK can find every feature without learning a navigation tree.

## iOS-specific notes

- The SDK is consumed via a local Swift Package reference (path:
  `../`). Edits to the SDK sources take effect on the next Cmd-R of
  the example app without a publish step.
- Deployment target is iOS 16 to match the SDK. The `.onChange`
  modifier uses the single-parameter (pre-iOS 17) form for that
  reason.
- API-key handling on iOS would idiomatically use an `.xcconfig`
  with a build setting injected into `Info.plist`. We kept the
  const-in-Swift pattern here for parity with the React Native
  sample; the trade-off is a less-idiomatic-iOS approach in exchange
  for one consistent how-to-replace-the-key story across the four
  apps.
